2 adapters: LocalStorageStore, InMemoryStore (tests)

Problem

Persistence plumbing (debounce timer, localStorage key, won-gate) is inline in 7 Game methods. Every mutation must call persist() — easy to forget, hard to test.

Solution

Extract a GameStore seam: save(GameSnapshot), load(): GameSnapshot | null, clear(). Game calls store.save() after mutations. The timer and localStorage key live inside the adapter.

Wins

  • Locality: persistence bugs live in one module
  • Test seam: InMemoryStore → no localStorage mocking
  • Interface shrinks: Game loses 4 methods, 1 flag, 1 timer
  • Persistence strategy swaps without touching Game
Note: The if (browser) block at line 603 would also move into the adapter: GameStore.hydrate(game) on import.

Problem

Three shallow modules each own a fragment of "how a card looks." Adding a new display concern (e.g. card border colour for highlight) means editing 3 files. The rank/suit maps are duplicated between card-images.ts and Board.svelte.

Solution

Merge into one display.ts module exportig 6 functions. All maps, glob imports, and image cache become private internals. Board.svelte imports just what it needs from one place.

Wins

  • Locality: card display logic in one module
  • Deletion test: deleting it concentrates complexity in one place, not 3
  • Interface shrinks: one import path instead of 3
  • No more duplicate rank/suit maps

Problem

Board.svelte imports 4 solver-internal modules to answer one question: "are there any legal moves?" It also duplicates Game.snapshot() because it's private (line 103). Every caller that needs solver interaction must know about generateMoves, deepClone, and GameSnapshot.

Solution

Add canProceed(): boolean to Game (or expose snapshot() publicly). The solver host module (or Game itself) answers "are we stuck?" without leaking move generation internals. solverMoveToHint() moves from Board into the solver or a bridge module.

Wins

  • Leverage: Board stops importing 4 solver-internal modules
  • Interface: Game.snapshot() going public removes the duplicate
  • Locality: "stuck" heuristic lives in one place
  • Testable: canProceed() is one assertion instead of 4-import orchestration

Problem

AnimationHost calls Game methods at arbitrary points during animation. Understanding the deal flow requires reading both files simultaneously. startSolve mutates game state step-by-step while animating — hard to debug, hard to test.

Solution

Mutate game state first, producing an AnimationPlan[] (from/to rects, card images, timing). AnimationHost receives the plan and only plays it back. No game state references during playback.

  • Leverage: AnimationHost can be tested with synthetic plans, no game state
  • Locality: all game-state mutation lives in Game
  • Debug: plan is inspectable — replay or skip at will
  • Trade-off: requires computing card rects from state alone (already possible via pileRect/pileCascade in AnimationHost)
Caveat: This is the largest refactor of the four. The deal animation iterates 28 cards, and the solve animation loops through unknown move count. The plan must account for cascade changes after each card removal (face-up flips). Worth prototyping in a branch before committing.

Top recommendation

Extract persistence behind a seam

It's the highest-leverage, lowest-risk change. The persistence code is already self-contained (lines 452–503, 603–624 in game.svelte.ts) — extracting it into a GameStore module with a clear interface removes 4 methods and 1 state flag from the overloaded Game class, creates a real test seam (localStorage vs in-memory), and requires no behavioural changes. It's a pure extraction, not a redesign. Do this first; it makes the Game class smaller and the remaining candidates easier to reason about.
